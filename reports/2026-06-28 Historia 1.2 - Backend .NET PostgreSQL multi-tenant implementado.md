## Decisiones técnicas clave

### 1. Dual-context EF Core — evitar conflicto de migrations

EF Core 10 aplica ALL migrations del assembly cuando se llama `MigrateAsync()` en cualquier DbContext. Con `AppDbContext` y `TenantDbContext` en el mismo assembly, ambos compartiría el mismo historial de migrations.

**Solución adoptada**: EF Core migrations SOLO para `AppDbContext` (public.tenants). El schema de tenant se crea con raw SQL en `ApplicationBuilderExtensions.EnsureTenantSchemaAsync()`:
```sql
CREATE SCHEMA IF NOT EXISTS "tenant_{id}";
CREATE TABLE IF NOT EXISTS "tenant_{id}".users (...);
CREATE UNIQUE INDEX IF NOT EXISTS ix_users_username ON ...;
```

### 2. TenantDbContextFactory — sin dependencia de ITenantContext

`AuthService` crea un `TenantDbContext` para verificar credenciales **antes** de emitir el JWT, así que no puede depender de `ITenantContext` (que se llena después del JWT). El factory appenda el schema directamente al connection string:

```csharp
var tenantConnStr = connStr.TrimEnd(';') + $";Search Path=\"{schemaName}\",public";
```

Npgsql interpreta `Search Path` en el connection string como `SET search_path` automático.

### 3. Dos mecanismos de search_path coexisten

- **TenantSchemaInterceptor** (DI): Para el `TenantDbContext` inyectado en requests HTTP autenticados. Ejecuta `SET search_path` en `ConnectionOpenedAsync`.
- **TenantDbContextFactory** (connection string): Para AuthService (login) y migrations de startup.

### 4. Seed parametrizado con BCrypt

El hash de BCrypt contiene caracteres `$` que rompen string interpolation en SQL. Solución con parámetros posicionales de EF Core:
```csharp
await publicCtx.Database.ExecuteSqlRawAsync(
    "INSERT INTO ... VALUES ({0}, {1}, {2}, {3}) ON CONFLICT DO NOTHING",
    userId, "admin", passwordHash, tenantId);
```

## Items pendientes para code review

- **Verificación CI**: `dotnet build` y `dotnet test` no se pueden correr localmente (sin SDK). La verificación se hace en GitHub Actions (`backend-ci.yml`).
- **TenantDbContextFactory.Create() — set search_path vía ExecuteSqlRaw**: El `SET search_path` llamado en `Create()` antes de retornar el contexto funciona si la conexión ya está abierta. En algunos scenarios EF Core podría abrir la conexión lazy. Revisar si se necesita un `EnsureOpenAsync()` antes.
- **AuthServiceTests — dispose doble**: El mock `tenantCtx` es disposed por `AuthService` (via `using var`) y nuevamente en el test. `DbContext.Dispose()` es idempotente así que no falla, pero es ruido. Candidato de limpieza.
